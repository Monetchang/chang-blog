过去一年，很多团队都在做同一件事：想办法把 token 账单压下来。

有人在做 prompt cache，有人在压缩上下文，有人在优化模型路由，也有人在减少不必要的多轮调用。可这些优化，基本都还停留在同一个前提里：推理依然发生在云端，成本依然按调用计费。

Gemma 4 的出现，让这个前提第一次开始松动。它不是一条普通的“又发布了一个开放模型”的新闻，而是一种更值得注意的信号：模型开始被认真地往设备侧、边缘侧和本地应用里推进。官方把 Gemma 4 定位为目前最强的 Gemma 开放模型，强调它面向 advanced reasoning 和 agentic workflows；同时，整个家族覆盖从移动和边缘设备到消费级 GPU、开发工作站的部署场景。

也正因为如此，“0 token 时代来了”这个说法才会迅速流行起来。

但这句话最容易被误读。

它真正的意思，不是 AI 从此免费，也不是云端 API 很快失去价值，而是：**有一部分原本默认应该发送到云端的请求，现在第一次有了现实的本地化可能。**

---

# 一、先回答一个基础问题：Gemma 4 到底是什么？

如果不先回答这个问题，后面关于“0 token”的讨论其实很容易飘。

Gemma 4 本质上是一组开放权重模型，但它和很多只适合研究、测试或局部实验的开放模型不太一样。它一开始瞄准的就不只是“跑榜”和“开源社区热度”，而是更明确的落地路径：**既能在本地设备上跑，也能接入更复杂的开发与产品流程。** 官方在发布时强调，它面向 advanced reasoning、agentic workflows 和 intelligence-per-parameter；Hugging Face 侧也把它概括为“frontier multimodal intelligence on device”。

更重要的是，它不是只会“聊天”的那种本地模型。围绕 Gemma 4，配套出现的是一整条更完整的端侧链路：设备端体验应用、Agent Skills、多步工作流、LiteRT-LM、CLI，以及跨 Android、iOS、桌面、Web、IoT 和 Raspberry Pi 的部署路径。官方甚至把“直接在设备上运行多步 autonomous workflows”当成重点来展示。

这意味着，Gemma 4 真正值得关注的地方，不只是“参数和榜单”，而是它在试图回答一个过去总是停留在演示阶段的问题：

> 模型不只是能不能在设备上运行，而是能不能在设备上完成一部分真实任务。

从这个角度看，Gemma 4 更像一个分界点。它不一定已经把端侧 AI 的所有问题都解决了，但它确实把“设备端 AI”从概念讨论推到了工程讨论。

---

# 二、为什么 Gemma 4 会和“0 token”联系在一起？

因为它第一次让一部分请求“有理由不走云端”。

所谓“0 token”，并不是一个严格的技术术语，更像是一种行业表达。它想描述的是：**如果推理完全在本地完成，不再调用远程 API，那么这些请求在账单层面就不再产生按 token 计费的成本。**

过去，大模型应用几乎默认建立在云端推理之上。无论你做多少缓存、压缩多少上下文、设计多复杂的路由，只要核心推理仍然发生在远端，token 成本就始终存在。Gemma 4 把另一种可能性真正推到了眼前：对于一部分高频、短上下文、对联网依赖不强、对隐私较敏感的任务，本地推理开始变得值得认真考虑。官方对小模型的表述也很明确：它们优先强调的是 on-device utility、低延迟、多模态能力和生态集成，而不是单纯堆参数。

媒体和社区之所以会迅速把这件事和“0 token 时代”挂钩，也正是因为本地体验已经不再只是“能跑起来”。有报道提到，开发者在 iPhone 17 Pro 上通过针对苹果芯片优化的 MLX 跑出了超过 40 token/s 的速度；同时，官方提供的设备端应用也让非极客用户能更直接地体验本地运行。

所以，“0 token”并不是一句空话。它有现实基础。

但它也绝对不能被理解成“0 成本”。

---

# 三、0 token 不是没有成本，而是成本结构发生了变化

这是这篇文章最想说清楚的一点。

过去你为云端付费，买到的是一整套已经被平台封装好的能力：远程推理、弹性扩容、基础设施维护、可用性保障、监控告警、跨区域分发，以及持续的模型服务更新。

当一部分请求被迁回本地，这些成本并不会消失，只会以另一种形式重新出现：设备性能占用、内存与存储压力、功耗与发热、模型分发、版本兼容、失败回退、质量评估、终端碎片化适配，以及整个本地推理链路的工程维护。Gemma 4 的意义，不是让行业突然进入“免费 AI”，而是让越来越多团队开始认真计算：哪些任务值得把云端可变成本换成本地设备成本和工程成本。这个判断也与官方持续强调的 on-device deployment、structured output 和 agentic runtime 路线相一致。

所以，与其说 Gemma 4 让行业进入了“0 token 时代”，不如说它把行业从“按 token 优化”推进到了“按任务分层”。以前大家优化的是“这一轮对话多少钱”，接下来更值得优化的是“这一类任务到底该不该默认送上云端”。

---

# 四、Gemma 4 真正改变的，不是模型强弱，而是部署重心

过去几年，开源模型当然一直在进步，但很长一段时间里，“本地模型”的主流印象仍然是：

* 能跑，但不够快
* 能看 demo，但不够稳
* 能聊天，但很难接进真实工作流
* 能做点辅助，但离生产可用还有明显距离

Gemma 4 至少在方向上把这件事往前推了一步。

一方面，它把“多模态 + 端侧 + agentic workflow”三件事放在了同一个叙事里。另一方面，生态也在明显围绕“deploy anywhere”展开：Hugging Face 侧强调 Apache 2.0、multimodal、on-device，以及与 transformers、llama.cpp、MLX、WebGPU、Rust 等常见工具的整合；Google 则同步强调移动端、桌面端、IoT、Raspberry Pi、Android AICore 和 LiteRT-LM 的部署路径。

这说明一个趋势：**模型的竞争，正在从“谁只能跑在云上”转向“谁能更自然地分布在本地、边缘和云之间”。**

这件事一旦成立，产品和架构的默认选项就会变。未来很多团队在设计 AI 功能时，第一反应不再是“接哪个云 API”，而会先问一句：这类请求，本地能不能接住？

---

# 五、最先被本地吞掉的，不会是最复杂的任务

每次技术边界变化，大家都会本能地高估它对复杂场景的替代速度。

端侧推理也一样。

Gemma 4 真正最先改写的，不会是那些超长上下文、复杂工具编排、强结构化输出、强联网依赖、强企业治理要求的任务。最先被迁回本地的，通常会是另一类请求：

* 高频、短请求
* 逻辑相对稳定
* 容错空间较大
* 不强依赖实时联网
* 对隐私、本地响应和离线能力更敏感

比如日常问答、轻量写作辅助、短文本总结、基础图像理解、本地知识整理、简单结构化生成，甚至部分低风险的设备控制或工作流触发。Gemma 4 官方围绕设备端 utility、agent skills 和多步本地工作流的展示，本身就更贴近这类任务，而不是“一切都本地化”的激进设想。

这也是为什么“0 token”这个说法成立，但成立得并不全面。它最先对应的，不是“所有 AI 请求都不需要云”，而是“有一部分过去默认用云的请求，现在值得被重新分层”。

---

# 六、复杂 Agent 为什么短期内依然离不开云？

因为复杂任务消耗的从来不只是模型参数，而是整个系统能力。

这一点，从社区的实际反馈里也能看出来。有体验者在更强硬件上运行 26B A4B 时发现，直接对话和代码解释都可以很顺，但一旦真的把它当成 coding agent 去用，问题就开始集中暴露：大上下文、复杂 prompt、稳定工具调用、结构化输出，这些要求叠加起来之后，模型会出现卡住、报错或输出结构不对的情况。

这背后恰恰说明了一件事：**“本地能跑模型”不等于“本地能稳定跑复杂 Agent”。**

复杂 Agent 真正要求的，通常是以下几种能力同时成立：

* 长上下文处理要稳定
* 工具调用要可靠
* 输出结构要可控
* 多步执行要可观测
* 失败后要有恢复与回退机制
* 还可能需要联网、权限系统和外部资源访问

这些问题里，很多都不是模型本身单独能解决的，而是推理引擎、运行时、工具链、服务层和治理层一起决定的。

所以，至少在未来一段时间里，端侧模型更适合先吃掉“高频但相对轻”的任务；而复杂 Agent、企业级协作流程和高 SLA 场景，依然更适合云端承担主力。

---

# 七、这对云端 API 意味着什么？

不是消失，而是重新定义。

如果设备端越来越能完成基础和高频任务，那么云端 API 继续收费的理由，就不能只建立在“我这里有个模型接口”之上。它必须证明，自己卖的不只是推理本身，而是一整套本地难以替代的系统能力。

这一点，其实已经体现在云端服务的产品形态上了。Gemini API 在同一时间引入了 Flex 和 Priority 这样的服务层级，本质上就是把“成本”和“可靠性”拆成不同档位来卖。

这说明未来云端真正有价值的部分，会越来越集中在这些能力上：

* 更高的可靠性和 SLA
* 更强的峰值弹性
* 更稳定的长上下文处理
* 复杂工具与外部系统编排
* 企业级权限、审计与合规
* 联网、实时数据与跨系统集成

所以，Gemma 4 对云端 API 的冲击，不是“云端没用了”，而是“纯 token 搬运型 API 的价值会被压缩”。

以后云端更像卖两件事：
第一，**本地做不到或做不稳的能力**；
第二，**本地不值得自己维护的能力**。

# 八、放到国内市场看，Gemma 4 和“0 token”真正改变的，其实不是“免费”

如果把 Gemma 4 和“0 token”的讨论原样搬到国内市场，其实会有一点错位。

原因很简单：在国内，很多国民级大模型应用对普通用户本来就是免费的。无论是豆包，还是元宝，C 端用户在 App 里直接使用时，感知到的都不是“按 token 付费”，而是“打开就能用”。从用户体验层面看，这本身就已经很接近“0 token”了。应用宝上的官方介绍显示，豆包已经把问答、搜资料、写代码、生成文档和 PPT、P 图、图片和视频生成、语音输入与语音通话等能力整合进一个免费应用里；元宝也在突出深度思考、联网搜索、文档精读、代码运行、图片编辑、视频生成、语音通话，以及与微信、腾讯文档、腾讯新闻、微信读书等生态的联动。

所以，Gemma 4 或“0 token”对国内市场的第一层影响，并不是“让终端用户第一次不用为 token 付钱”。这件事，国内头部 AI 应用其实早就做了。国内市场更早进入的是一种“前台免费、后台消耗巨量词元”的竞争阶段：用户不付费，不代表 token 不存在，只是费用被平台吸收了。国家数据局公开信息显示，到 2026 年 3 月，国内日均词元调用量已经超过 140 万亿，相比 2024 年初增长超过千倍。也就是说，国内大模型应用虽然在用户侧看起来接近“0 token”，但在平台侧，token 经济学不但没有消失，反而变得更重要了。

这意味着，Gemma 4 对国内市场真正重要的影响，首先不是“免费化”，而是**重新定义免费背后的成本结构**。过去国内厂商已经通过超低价 API 和免费 App 把用户侧价格压到了接近零。公开报道显示，字节曾在 2024 年把豆包主力模型价格压到 0.0008 元/千 Tokens，并把这轮竞争直接推成行业价格战；到了 2025 年底至 2026 年，国内 AI 原生 App 的竞争又进一步演化成以免费获客、生态导流和高频使用为核心的入口争夺。QuestMobile 公布的 2025 年 12 月数据里，豆包月活达到约 2.27 亿，元宝也进入头部梯队。

换句话说，国内市场不是在等“0 token”到来，而是早就处在“用户免费”的环境里，只是过去这套免费更多依赖云端补贴、低价推理和平台兜底。Gemma 4 让大家开始认真思考的是：**这部分成本，是否可以有更多比例从云端服务成本转移到设备侧算力与端侧执行。**

从这个角度看，Gemma 4 对国内市场的第一个真正影响，是把竞争焦点从“免不免费”进一步推向“值不值得留在云上”。因为免费在国内已经不是差异化，**更低延迟、更强隐私、更稳定的离线体验、更自然的设备集成**，才可能成为下一阶段差异化。尤其在国内，头部 AI 应用本身就不是孤立产品，而是背靠内容、社交、办公、支付、搜索和硬件生态。Gemma 4 这类面向端侧、多模态、可嵌入工作流的开放模型，对国内厂商的启发，不只是“能不能做一个本地聊天 App”，而是“能不能把 AI 更深地塞进手机、耳机、输入法、文档、浏览、搜索、拍照、学习和内容创作这些高频原生入口里”。

第二个影响，是它会进一步强化国内市场“入口战”而不只是“模型战”。既然用户端定价已经接近零，那么真正关键的，就不是谁先把价格降下来，而是谁能把免费 AI 更深地嵌进既有流量入口和既有产品关系里。Gemma 4 提供的不是直接答案，但它强化了一个方向：**未来国内市场的优势，不一定只来自“我有一个更强的大模型 App”，而可能来自“我把足够好的模型能力塞进了最多人、最高频、最原生的使用场景里”。** 头部 AI 原生 App 的高活跃数据，本身就说明入口和留存比“是否免费”更能拉开差距。

第三个影响，反而更偏 B 端和平台侧。国内 C 端用户虽然习惯免费，但平台并不可能无限承受所有调用都走云端的成本压力。尤其当 AI 从对话走向搜索、写作、图片、视频、代码、文档、录音、代理执行后，调用密度和多模态负载都会显著上升。Gemma 4 这类可在终端、边缘设备和本地硬件上部署的模型，对国内厂商的现实意义之一，就是给“平台承担全部推理成本”这件事提供了新的优化空间：把一部分高频、轻量、隐私敏感、离线有价值的任务下沉到端侧，把云端集中留给复杂推理、联网检索、多步 agent 和企业级能力。这对国内厂商来说，不只是技术路线变化，更是商业模式上的再平衡。因为在“用户免费”已经成为常态的市场里，谁能把后台成本打得更低、把端云分工做得更合理，谁就更有机会把免费能力长期维持下去。

所以，Gemma 4 和“0 token”对国内市场的真正意义，不是让国内用户第一次享受到“免费 AI”，而是把国内大模型竞争从“免费供给”进一步推向“免费之上的效率竞争”。在海外，“0 token”首先冲击的是用户付费和 API 计费心智；在国内，它更深层的影响其实是三件事：**把平台侧成本重新分配、把竞争重心从独立 App 推向原生入口、再把产品差异化从“能不能免费”推向“能不能更快、更稳、更私密、更无感地融入日常场景”。**

# 九、更现实的架构，不是全本地，而是“本地优先，云端兜底”

很多讨论一谈到端侧模型，就喜欢走向两个极端：

* 要么觉得本地完全不现实
* 要么觉得未来一切都应该本地化

真正成熟的工程方案，通常不在这两个极端里。

更现实的路径，大概率是：**本地优先，云端兜底。**

也就是说，先把那些最适合本地化的请求迁到设备侧：

* 高频
* 短上下文
* 容错空间大
* 隐私敏感
* 离线有价值
* 对延迟敏感

然后把剩下那些更复杂、更长、更依赖联网、更要求稳定性的任务继续留给云端。与此同时，再加上一层清晰的路由规则：

* 本地能处理时优先本地
* 超过复杂度阈值自动升级到云端
* 本地失败时自动 fallback
* 涉及敏感数据时优先不出端

这样的好处是，它不是靠信仰选边，而是按任务特征做分层。

对产品团队来说，这种架构通常能同时带来三种收益：一部分云端成本下降，一部分交互延迟变短，一部分敏感数据可以留在本地不出域。对工程团队来说，它也比“All in 本地”更稳，因为你不需要一次性把所有问题都解决，只需要先解决最容易回本的那一层。

---

# 十、现在真正值得做的，不是争论“要不要本地化”，而是先回答三个问题

## 1. 你的请求结构到底是什么？

哪些请求最高频？哪些请求最贵？哪些请求对隐私最敏感？哪些请求对联网依赖最低？只要把这几个维度拉出来看，通常就会很快发现，真正值得迁移的不是全部，而是其中一小部分高频任务。

## 2. 哪些任务迁到本地最容易回本？

通常不是最炫的那类，而是最稳定、最重复、最标准化的那类。调用量高、上下文短、输出相对固定、结果偏辅助而不是最终决策，这样的任务往往最适合先迁。

## 3. 你的系统有没有能力做“分层路由”？

未来最合理的 AI 架构，未必是一个“只有本地”或“只有云端”的系统，而更像一个能根据任务复杂度、上下文长度、设备能力、隐私等级和服务要求自动做选择的混合系统。

如果一个团队今天就开始从这三个问题入手，那么 Gemma 4 带来的就不只是一次模型热点，而会真正变成一次架构升级的起点。

---

# 结语

Gemma 4 并没有在一夜之间把行业带进一个彻底的“0 token 时代”。

但它确实做成了一件更重要的事：它让“有些 AI 请求根本不该默认去云端”这件事，第一次变得足够具体、足够现实、也足够值得认真设计。官方给出的模型能力、端侧工具链和设备部署路径，说明这已经不是一个只能写在趋势报告里的想法，而是在开始变成可以设计、可以实验、可以落地的系统选项。

它带来的真正变化，不是简单地让 token 变少了，而是让我们开始重新思考 AI 系统的边界：
哪些能力该留在设备上，哪些能力继续交给云；哪些任务该追求本地即时响应，哪些任务仍然需要远程的稳定性、弹性和治理能力。

未来几年，AI 基础设施的主线，很可能不再只是“谁的模型更强”，而是“谁能更合理地定义本地、边缘与云端的分工”。

如果说过去大家优化的是一次调用的价格，那么接下来更重要的，可能是重新定义：**哪些调用本来就不该发生。**
