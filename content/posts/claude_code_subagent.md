最近一段时间，Claude Code 讨论度很高。它之所以值得认真看一眼，不只是因为它是一个“会写代码的 AI 工具”，而是因为它把很多人做 Agent 时迟早会遇到的结构问题，提前摆到了台面上。

Anthropic 对 Claude Code 的定位很直接：它不是普通补全工具，而是一个 agentic coding tool，能够读代码库、编辑文件、运行命令，并和开发环境配合完成任务。官方对 subagents 的描述也很明确：subagents 是专门处理特定任务类型的 AI 助手，运行在各自独立的 context window 中，可以拥有自定义 system prompt、特定工具访问和独立权限；Claude Code 在合适情况下还会自动把任务委派给它们。

这件事看起来像一个功能点，实际上背后对应的是一个更大的工程判断：

> 当任务复杂度上来之后，问题往往已经不再是“模型够不够聪明”，而是“系统结构对不对”。

我想写这个系列，也是因为这一点。  
很多 Agent 项目做不稳，不是因为团队没调好 prompt，也不是因为模型不够新，而是因为一开始就默认：一个 Agent 应该尽可能什么都做。Claude Code 的 subagents 恰好提供了一个很好的切口，让我们重新讨论这个默认前提到底对不对。

---

# 一、为什么从 Claude Code 讲起，而不是直接讲“多 Agent”概念

如果只是抽象地讨论多 Agent、delegation、context isolation，这些词都不难理解。但问题在于，抽象概念很容易讲成空话。Claude Code 的价值，在于它给了一个很具体的产品化落点：当一个工具真的需要在代码库里持续读写、执行、分析、回看上下文并完成多步骤任务时，单体 Agent 会天然撞上什么问题？官方为什么没有继续把所有能力都压进一个“大而全”的主 Agent，而是要引入 subagents？

我觉得这个切口很有代表性，因为代码场景对 Agent 的要求特别苛刻。

它不是单轮问答。  
也不是一次性内容生成。  
它要面对的是一个会持续膨胀的任务环境：

- 代码库本身很大  
- 用户要求往往是多步骤的  
- 中间会穿插文件读取、命令执行、测试反馈  
- 任务链路可能很长，而且经常要反复修改  

在这种环境里，如果仍然坚持“一个 Agent 统筹一切、处理一切、记住一切”，那系统很快就会表现出一种大家都熟悉但又说不太清的症状：看起来什么都能做，但越往后越不稳。

Claude Code 里 subagents 的存在，本质上就是在公开承认这件事：

> 上下文不是无限的，角色也不该是混在一起的。

这也是我觉得它值得单独拿出来写的原因。它不是“又一个多 Agent 框架”，而是把一个很多团队已经碰到、但还没有系统表达清楚的问题，直接产品化了。

# 二、单 Agent 的诱惑很大，但它的问题其实是结构性的

刚开始做 Agent 的时候，几乎所有人都会先走到单 Agent 这一步。这很正常，因为它足够符合直觉。

用户给一个目标，Agent 自己理解任务、自己调用工具、自己总结结果。链路非常漂亮，展示也很好看。只要模型能力在线，再加上一点工具和 prompt 调整，前几次体验往往不差。

问题是，这种“前几次体验不差”，很容易让人误以为方向就是对的。

实际上，单 Agent 在复杂任务里暴露出来的很多问题，不是偶发，而是结构性必然。

## 1. 上下文会越来越脏

Claude Code 官方强调 subagents 的一个核心价值，就是 improved context management，以及让主会话保持更干净。这个表述其实已经点到问题本质了：复杂任务不是只靠更大的上下文就能自然解决的，反而越长的链路越容易把上下文搅浑。

一个单体 Agent 每多做一步，都会往自己的上下文里塞更多东西：

- 历史对话  
- 中间推理  
- 工具返回  
- 临时约束  
- 失败后的修补说明  

刚开始这些信息看上去都“有用”，但任务一长之后，它们并不会自动分层。模型看到的是一整团历史痕迹，而不是一个被干净整理过的任务状态。

在代码场景里，这种问题尤其明显。因为它不只是“聊天记录变长”，而是文件上下文、命令结果、报错日志、修复策略都会累积。单 Agent 在早期也许还能勉强扛住，但随着链路变深，很多漂移都来自上下文已经不再干净。

Claude Code 把 subagent 放到独立 context window 里，本质上就是在结构层面承认：  
**有些任务不该继续污染主线程。**

## 2. 角色会开始打架

官方对 subagents 的设计，强调的是 task-specific workflows，也就是任务类型明确、角色相对单一的工作流。换句话说，它不是在追求“一个更大的万能脑”，而是在强调“专门角色处理专门任务”。

这点特别重要。

因为很多单 Agent 失控，不是它不知道答案，而是它在不同角色之间来回切换：

- 一会儿像规划者  
- 一会儿像执行者  
- 一会儿像代码审查者  
- 一会儿又像文档解释员  

这些角色放在一个人类团队里，本来也常常会拆开。可在很多 Agent 设计里，它们却被直接塞进同一份系统提示词和同一个上下文里。任务短的时候问题不大，任务一长，角色中心就开始漂。

Claude Code 为什么会让你通过 `/agents` 查看、创建和管理 subagents，并且支持根据任务类型自动委派？很大一部分原因，就是它把“角色”当成了需要显式设计的对象，而不是临时在 prompt 里凑出来的语气切换。

## 3. 单体 prompt 会逐渐变成黑盒

这点虽然官方不会直白地说，但做过 Agent 的人都知道。

当系统把越来越多职责塞给一个主 Agent 时，你迟早会开始往 prompt 里不断打补丁：

- 加一条规则，防止它乱改  
- 加一个例子，纠正它理解偏差  
- 加一段边界说明，避免它误调用工具  
- 再加一段失败后的处理方式  

最后你得到的，通常不是一个更优雅的系统，而是一段越来越长、谁都不敢轻易改的黑盒配置。

Claude Code 的 subagents 至少提供了另一条路：  
不是继续把复杂度往一个主提示词里堆，而是把复杂度拆出去，让不同角色各自管理自己的任务上下文和职责。

# 三、Claude Code 给出的不是“技巧”，而是一种结构判断

我觉得 Claude Code 最值得学的地方，不是它“支持 subagents”这件事本身，而是它隐含的一种工程判断：

> 复杂任务应该先拆结构，再谈增强能力。

这听起来像一句正确的废话，但实际上很多 Agent 项目恰恰是反着做的。

系统一不稳定，第一反应是：

- 换更强模型  
- 扩更大上下文  
- 补更多提示词  
- 加更多工具描述  

这些动作有时也有效，但它们大多是在增强一个单体，而不是重构一个系统。

Claude Code 这里做的事刚好相反。  
它一边强调 subagents 带来更好的 context management，一边明确提示：subagents 适合并行任务、独立工作流、需要隔离上下文的场景；对于简单任务、顺序性很强的操作、或必须跨步骤维持连续上下文的情况，则不一定要委派。

这说明它并不是在鼓励“能分就分、越多越好”，而是在表达一个更成熟的观点：  
**什么时候该拆，什么时候不该拆，本身就是架构设计的一部分。**

这恰恰比“多 Agent 更高级”这种口号更重要。

---

# 四、Subagent 的重点，不是“多”，而是“边界清晰”

如果只从字面理解，subagent 很容易被误读成“主 Agent 再派几个小弟去干活”。但从 Claude Code 的定义看，subagent 不是简单分身，而是一个带有独立 context、自定义 system prompt、特定工具访问和独立权限的专门助手。这里面最重要的关键词，其实不是“agent”，而是“专门”“独立”“特定”。

这意味着一个好的 subagent 设计，不应该追求全能，而应该追求边界清晰。

比如在代码场景里，你完全可以想象这样的拆法：

- 一个角色专门负责代码探索与信息收集  
- 一个角色专门负责测试与失败归因  
- 一个角色专门负责安全审查  
- 一个角色专门负责修改建议整合  

它们不需要共享所有过程细节，也不需要都看到完整主线程历史。  
主线程只需要在合适的时候把合适的问题交出去，再把结果拉回来。

这种设计比单体 Agent 最大的变化，不在于“能力更强”，而在于它开始出现组织秩序了。

这也是为什么 Claude Code 的相关机制里会区分不同 subagent 的身份，因为一旦系统真的进入多角色协作，区分“是谁在做什么”就不再是个小问题，而是可观测性的基础。

---

# 五、为什么说 Delegation 才是这件事的核心

很多文章谈 Sub-agent，容易把重点放在“定义几个角色”上。但角色本身只解决了一半问题。另一半更关键的问题是：**任务怎么在这些角色之间流动？**

Claude Code 给了两个很有代表性的信号：

第一，它支持自动 delegation，也就是 Claude Code 会在合适时把任务交给专门的 subagent。  
第二，它也允许你显式指定或管理 subagents，比如通过 `/agents` 查看和使用。

这说明系统设计的重点，并不只是“存在若干角色”，而是“存在一个合理的委派机制”。

Delegation 真正解决的是主 Agent 的定位问题。  
主 Agent 不应该继续做那个什么都亲自下场的全能角色。它更像调度者：

- 识别当前任务属于哪一类  
- 判断是否值得委派  
- 选择哪个 subagent 更合适  
- 把结果收回来并继续主流程  

当你这样理解主 Agent 时，整个系统会一下子变得顺很多。  
因为主 Agent 的任务不再是“记住全部细节并亲自完成全部细节”，而是维持整体秩序。

我觉得这是 Claude Code 这套设计最容易被低估的地方。  
很多人看到了 subagents，却没有进一步看到：subagents 之所以成立，是因为主线程终于不再把自己当成万能执行者了。

# 六、借 Claude Code 这个例子，也能把几个常见边界讲清楚

## 1. Sub-agent 和 Tool 不是一回事

Claude Code 的 subagents 定义里明确提到，每个 subagent 可以拥有特定工具访问。这个表述本身就说明：subagent 和 tool 是两层东西，不是同一个概念。

Tool 更接近确定性执行单元。  
比如读文件、运行命令、查文档、调接口。它们要做的是稳定、明确、可重复。

Sub-agent 更接近策略性角色。  
它负责理解任务、选择动作、组织若干工具的使用，并把结果整理成适合主线程继续处理的输出。

很多系统一开始就把这两者揉在一起，导致该稳定的地方不稳定，该灵活的地方又不够灵活。  
Claude Code 的做法至少给了一个比较清楚的分层：角色拿着特定工具完成特定类型的工作，而不是把“能力”都混成一锅。

## 2. Sub-agent 也不是 Workflow 的替代品

官方也明确给了一个边界：并不是所有任务都需要 subagents。简单任务、顺序性强的任务、需要跨步骤保持连续上下文的任务，可以直接在主线程里完成。

这其实很像在说：subagent 擅长处理的是“需要动态委派、适合隔离上下文、可以独立完成的子任务”，但不意味着传统 workflow 或顺序控制没用了。

更合理的做法通常还是：

- 用 workflow 管住主流程阶段  
- 用 subagent 处理阶段内部的不确定性  

这样才不会一上来就把一切都交给动态决策，最后把系统变得既不好调试，也不好解释。

# 七、Claude Code 这个案例还能提醒我们另一件事：有 delegation，就一定要有 fallback

只要系统里出现委派，就一定会有委派失败的时候。

如果主线程把任务委派出去了，那它就必须准备好处理失败返回、异常输出、角色选错和结果冲突。否则 delegation 只会把复杂度从一个地方挪到另一个地方。

我觉得这也是很多团队容易忽视的点。

他们学到了“拆角色”，却没有同步补上“失败路径设计”。  
结果就是系统在 happy path 上看起来很漂亮，一旦子任务偏航，主线程就不知道该怎么接。

比较实用的思路通常还是三层：

第一层先校验。  
看子 Agent 返回的内容是否可接受，至少先把格式错、证据不足、明显不一致的情况拦下来。

第二层再重试，但不要只是原样重试。  
更有价值的是切换角色、切换策略或缩小问题范围。

第三层是降级。  
如果高级路径不稳定，那就回到更保守但更稳的 baseline。

这部分虽然不一定是 Claude Code 官方展开的重点，但恰恰是从 Claude Code 这种“显式 delegation”设计自然推导出来的工程结论：  
**一旦任务是分发出去做的，系统就必须认真设计收不回来的情况。**

# 八、为什么我觉得 Claude Code 值得被拿来当成一个系列的起点

写到这里，基本也能解释这篇文章为什么还要坚持把 Claude Code 放在标题里，而不是干脆改成一篇泛泛的 Agent 方法论。

因为它确实提供了一个足够具体、又足够典型的例子。

Claude Code 不是在抽象层面告诉你“多 Agent 可能更好”，而是在一个真实的高复杂度场景里，把几个关键判断同时落了地：

- 它是 agentic coding tool，而不是单纯补全器。  
- 它让 subagents 成为正式能力，而不是社区技巧。  
- 它把 subagent 定义成独立上下文、独立提示、特定工具访问、独立权限的专门角色。  
- 它同时强调：不是所有任务都该委派，subagents 有明确适用边界。  

这几件事放在一起，已经不是“某个功能体验不错”那么简单了。  
它更像是在告诉我们：一个真正可用的 Agent 系统，不能只靠模型能力堆出来，它必须开始像系统一样被设计。

这也是我想从它写起的原因。  
不是因为 Claude Code 是唯一答案，而是因为它把很多原本散落的工程直觉，压缩成了一个非常清晰的观察窗口。

---

# 结语

如果把全文压缩成一句话，我会这样说：

> Claude Code 值得研究，不是因为它有 subagents 这个新功能，而是因为它用 subagents 明确表达了一件事：复杂任务不能继续靠单个 Agent 硬撑，必须开始依赖结构化分工。

单 Agent 的问题，从来不只是“上下文不够大”或者“prompt 没写好”。  
很多时候，真正的问题是：我们把本该靠组织结构消化的复杂性，错误地塞进了一个越来越臃肿的上下文里。

Claude Code 在这里给出的启发很直接：  
先把角色拆开，把上下文隔离，把委派机制设计出来，再去谈能力增强。  
这条顺序，也许比“再换一个更强模型”更重要。

对我来说，这也是 Agent 工程化真正开始的地方。
