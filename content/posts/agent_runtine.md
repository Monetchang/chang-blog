> 代码生成会越来越便宜，但 Agent Runtime、验证闭环和工程判断会越来越贵。

最近 Andrej Karpathy 关于 **From Vibe Coding to Agentic Engineering** 的访谈，在技术圈又引发了一轮讨论。原访谈视频地址：https://www.youtube.com/watch?v=96jN2OCOfLs，大家有兴趣可以看看。

这类讨论每隔一段时间就会出现一次。

Copilot 火的时候，大家问：初级程序员是不是危险了？

Cursor 火的时候，大家问：以后是不是不用自己写业务代码了？

Claude Code、Codex、各种 Coding Agent 出来之后，问题又升级了：

> AI 不只是补全代码了，它开始能读代码库、改文件、跑命令、看报错，甚至尝试完成一个完整工程任务。那程序员的价值还剩下什么？

这个问题很容易被讨论成两个极端。

一种观点是，程序员很快会被 AI 替代。

另一种观点是，AI 只是工具，程序员完全不用担心。

但我觉得这两个判断都太粗了。

更准确的说法应该是：

> **AI 不会简单淘汰程序员，但会重新给程序员定价。**

如果一个程序员的核心价值只是“把需求翻译成代码”，那么这部分价值确实会被快速压缩。

但如果一个程序员的价值在于定义问题、设计系统、判断方案、控制风险、验证结果，那么 AI Coding 反而会放大这类人的价值。

这也是我看完 Karpathy 这次访谈后最大的感受：

> **Vibe Coding 只是开始，真正重要的是 Agentic Engineering。**
>
> 而 Agentic Engineering 的长期壁垒，不是模型多会写代码，而是 Agent Runtime 能不能让 AI 在真实工程环境里可控、可靠、可验证地工作。

---

# 01 AI 会写代码之后，程序员的价值还剩什么？

过去我们讨论程序员价值时，默认有一个前提：

代码是稀缺的。

你会写 Java、Go、Python、React、SQL，你能把需求变成系统，这就是你的价值。

但 AI Coding 正在改变这个前提。

现在，越来越多代码不再需要人一行一行手写。你可以把需求描述给 AI，让它生成页面、接口、脚本、测试、配置，甚至直接在项目里修改文件。

这当然不是说 AI 已经能完全替代一个成熟工程师。

但它确实改变了一个基本事实：

> **代码生成本身正在变便宜。**

一旦代码生成变便宜，程序员的价值就会被重新拆分。

过去，一个需求从想法到上线，中间有大量工作都被归类为“写代码”：

```text
理解需求
拆分模块
设计数据结构
编写接口
实现页面
处理异常
补充测试
上线验证
后续维护
````

但在 AI Coding 时代，这些工作会被重新分层。

其中一部分偏实现的工作，会被 AI 大量接管。

比如：

```text
生成样板代码
补齐接口逻辑
写单元测试初稿
解释报错
迁移简单代码
生成 CRUD 页面
写脚本处理数据
```

这些工作不是不重要，而是会越来越不稀缺。

真正稀缺的会变成另一类能力：

```text
这个需求该不该做？
这个方案是否合理？
这个抽象未来会不会拖累系统？
这个实现有没有安全风险？
这个结果怎么验证？
这个 Agent 到底能不能被信任？
```

所以，AI Coding 带来的不是简单的“程序员失业”，而是程序员价值结构的变化。

过去你值钱，是因为你能写出代码。

未来你更值钱，是因为你知道：

> **什么代码应该被生成，什么代码不应该存在，以及生成之后如何验证它是对的。**

这才是更大的变化。

# 02 Karpathy 的提醒：Vibe Coding 只是开始

Karpathy 这次访谈之所以值得关注，不是因为他又讲了一遍“AI 会写代码”，而是因为他把 AI Coding 分成了两个阶段：**Vibe Coding** 和 **Agentic Engineering**。访谈页面的章节也直接把 Software 3.0、Agents、Vibe Coding 等主题放在一起讨论，说明他的重点并不是单纯的代码补全，而是软件开发方式的整体迁移。

Vibe Coding 解决的是：

> 如何让 AI 帮我更快写代码？

它的典型方式是：

```text
我描述需求
AI 生成代码
我看结果
我继续反馈
AI 继续修改
```

这种方式非常适合原型开发、个人工具、小型项目和快速试错。

它的意义在于降低门槛。

以前不会写代码的人，现在也可能做出一个可用工具；以前需要几天搭出来的 demo，现在可能几个小时就能跑起来。

所以，Vibe Coding 提高的是软件创造的下限。

但 Agentic Engineering 解决的是另一个问题：

> 如何让 Agent 在真实工程环境中可靠地完成任务？

这就不是简单生成一段代码了。

它要求 Agent 能够：

```text
理解目标
读取代码库
拆解任务
调用工具
修改文件
运行命令
查看错误
修复问题
验证结果
总结变更
```

这已经不是“代码生成”问题，而是“工程执行”问题。

Karpathy 在相关总结中也明确区分了这两者：Vibe Coding 让更多人能通过描述意图创造软件，而 Agentic Engineering 是一种更专业的工程纪律，重点是协调有缺陷但强大的 Agent，同时保住正确性、安全性、审美和可维护性。

这句话非常关键。

因为它说明，Vibe Coding 和 Agentic Engineering 的区别，不只是工具更强了，而是责任边界变了。

Vibe Coding 更像是：

```text
人主导，AI 辅助。
```

Agentic Engineering 更像是：

```text
人设计系统，Agent 在系统内执行。
```

所以我认为：

> **Vibe Coding 改变的是人和代码的交互方式；Agentic Engineering 改变的是软件工程的执行方式。**

# 03 代码生成越便宜，工程判断越重要

很多人第一次使用 AI Coding 工具时，最强烈的感受是“快”。

以前要查文档、写样板、调接口、改样式，现在可以直接描述目标，让 AI 先生成一个版本。

这当然很爽。

但用久了之后，另一个问题也会变得越来越明显：

> AI 很擅长生成代码，但不一定擅长判断代码是否应该这样生成。

它可能写出能跑的代码，但结构很差。

它可能快速实现一个功能，但留下很多重复逻辑。

它可能为了完成当前任务，引入一个不必要的依赖。

它可能修复了一个 bug，却破坏了另一个边界。

它可能给出一个“看起来合理”的方案，但长期看是坏设计。

这也是 Karpathy 对 AI 生成代码保持谨慎的原因。Business Insider 对他近期观点的报道提到，他认为 AI 写出的代码仍可能臃肿、脆弱、重复，并且仍然需要人类在审美、判断、品味和监督上负责。

这并不矛盾。

AI 代码能用，和 AI 代码足够好，是两件事。

Vibe Coding 降低的是实现成本，不是判断成本。

也就是说：

> **AI 让“做出来”变便宜了，但没有让“做对”变便宜。**

甚至在某些情况下，AI 还会放大坏判断的后果。

因为过去你写代码慢，技术债积累也慢。

现在 AI 生成代码很快，如果你没有足够的判断力，坏抽象、重复代码、临时方案和安全隐患也会更快进入系统。

这就是 AI Coding 最容易被低估的风险：

```text
代码生成速度提高了，
但复杂度积累速度也提高了。
```

所以，未来程序员不能把自己的壁垒建立在“我比 AI 更会写代码”上。

这个方向会越来越难。

更现实的壁垒是：

> **我能不能让 AI 在一个可控、可靠、可验证的工程系统里工作。**

这就进入了真正的核心：Agent Runtime。

# 04 真正的新壁垒：不是 Coding，而是 Agent Runtime

如果说 Vibe Coding 的入口是 Prompt，那么 Agentic Engineering 的底座就是 Runtime。

这里说的 Runtime，不是狭义的语言运行时，比如 JVM、Node.js、Python Runtime。

我这里说的 Agent Runtime，是一整套支撑 Agent 可靠工作的工程机制。

它决定了 Agent：

```text
看什么上下文
调什么工具
记住什么经验
处于什么状态
如何验证结果
在什么边界内行动
失败之后如何恢复
什么时候必须让人介入
```

这才是 AI Coding 进入深水区之后真正的壁垒。

因为模型会越来越强，代码生成会越来越便宜，很多工具都会接入类似的底层模型能力。

到最后，真正拉开差距的，不一定是谁能生成更多代码，而是谁能让 Agent 在真实工程环境中稳定工作。

Agent Runtime 至少包含六个核心部分：

```text
Context Runtime
Tool Runtime
Memory Runtime
State Runtime
Verification Runtime
Governance Runtime
```

下面分别展开。

---

## 4.1 Context Runtime：上下文不是越多越好，而是越准越好

Agent 要完成任务，第一件事是拿到上下文。

但上下文不是越多越好。

很多人容易把大上下文窗口当成最终答案，觉得只要能把整个代码库、所有文档、所有历史对话都塞进去，Agent 就会变聪明。

实际不是这样。

上下文太少，Agent 不知道情况。

上下文太多，Agent 可能更混乱。

真正难的是选择。

Agent 需要知道：

```text
当前任务相关文件有哪些？
哪些历史决策仍然有效？
哪些错误日志值得参考？
哪些用户偏好必须带入？
哪些信息已经过期？
哪些内容应该摘要？
哪些内容应该丢弃？
```

这就是 Context Runtime 的价值。

它不是简单扩展上下文长度，而是帮助 Agent 在正确时刻拿到正确信息。

一个成熟的 Context Runtime，应该像一个熟悉项目的技术负责人。

它不会把所有资料一股脑丢给你，而是会告诉你：

```text
这次改动重点看这几个文件。
这个接口之前有兼容性问题。
这个模块不能随便引依赖。
这个测试命令能验证核心路径。
```

所以：

> **Context Runtime 的核心不是扩大上下文窗口，而是提升上下文选择能力。**

未来很多 Agent 系统的差距，可能就体现在这里。

同样是一个模型，有的系统给它一堆杂乱上下文，有的系统给它精准、结构化、可执行的上下文，最终效果会完全不同。

---

## 4.2 Tool Runtime：工具调用不是开放权限，而是受控执行

Agent 和 Chatbot 最大的区别之一，是 Agent 能调用工具。

它不只是回答问题，还能行动。

比如：

```text
读取文件
修改文件
运行命令
查看日志
调用 API
查询数据库
创建 PR
执行部署
```

工具让 Agent 从“会说”变成“会做”。

但工具越强，风险越大。

如果一个 Agent 能随便改文件、删数据、跑命令、访问生产环境，那它就不再只是助手，而是一个有真实权限的执行体。

所以 Tool Runtime 的重点不是把工具全部开放给 Agent，而是把工具包装成安全、可控、可审计的接口。

它至少要解决这些问题：

```text
哪些工具可以调用？
调用参数如何约束？
哪些操作需要人工确认？
工具失败后如何重试？
工具返回结果如何解析？
危险命令如何拦截？
操作日志如何保存？
```

举个例子。

让 Agent 运行测试，可以自动执行。

让 Agent 删除文件，可能需要确认。

让 Agent 修改生产数据库，必须有更严格的权限和审批。

让 Agent 发起部署，也许需要经过 CI、测试、审核、灰度这些流程。

这不是模型聪不聪明的问题，而是工程边界问题。

所以：

> **Tool use 不是把能力交给 Agent，而是把能力包装成 Agent 可以安全调用的接口。**

未来真正成熟的 Agent 产品，一定不是简单堆工具，而是认真设计工具权限、工具协议、工具反馈和工具审计。

---

## 4.3 Memory Runtime：记忆不是聊天记录，而是经验系统

Memory 是 Agent 系统里很容易被误解的一层。

很多人把 Memory 理解成“保存历史对话”。

但真正有价值的 Memory，不应该只是聊天记录，而应该是经验系统。

一个 Coding Agent 需要记住的不是所有对话，而是那些会影响未来决策的信息。

比如：

```text
这个项目不允许引入重型依赖。
这个团队偏好显式错误处理。
这个接口曾经因为兼容性出过问题。
这个用户喜欢先给方案再改代码。
这个测试命令之前经常因为环境变量失败。
这个模块不要直接改，需要先看生成模板。
```

这些信息如果能被正确沉淀，Agent 下次执行任务时就不再像一个新员工，而更像一个熟悉项目的协作者。

但 Memory 也有风险。

记错的 Memory 比没有 Memory 更危险。

如果 Agent 把临时方案记成长期规范，把失败尝试记成成功经验，把已经过期的项目结构当成当前事实，它后面的判断都会被污染。

所以 Memory Runtime 要解决的不是“记住更多”，而是：

```text
什么值得记？
用什么结构记？
什么时候取出？
什么时候更新？
什么时候遗忘？
如何避免错误记忆污染决策？
```

我很喜欢用一句话总结：

> **没有 Memory 的 Agent 每次都像新员工；Memory 混乱的 Agent 则像一个记错很多事情的老员工。**

所以，Memory 的价值不在于存储，而在于提炼。

它不是聊天记录仓库，而是 Agent 的经验系统。

---

## 4.4 State Runtime：复杂任务需要状态机，而不是自由聊天

Chatbot 可以自由聊天。

但 Agent 不能永远自由行动。

一个复杂工程任务，不应该完全依赖模型“下一步想做什么”。

它应该有状态。

比如：

```text
理解需求
制定计划
等待确认
执行修改
运行验证
修复错误
总结变更
请求反馈
完成归档
```

State Runtime 的作用，是让 Agent 始终知道自己处于哪一步。

它要回答：

```text
现在任务阶段是什么？
下一步允许做什么？
什么条件下可以继续？
什么条件下必须停止？
什么时候需要人类确认？
什么时候应该回滚？
什么时候应该总结经验？
```

这听起来有点传统工程味，但越是强大的 Agent，越需要状态机。

因为 Agent 越能干，如果没有状态约束，它的行动空间就越大，也越容易偏离目标。

很多 Agent 失败，不是因为模型完全不会做，而是因为执行过程没有边界。

它可能在错误方向上不断尝试。

它可能修一个 bug 修着修着开始重构整个模块。

它可能为了通过测试引入一个更大的隐藏问题。

状态机的价值，就是限制这种随机游走。

所以：

> **Agent 越自主，越需要状态约束。**

复杂任务不是靠自由聊天推进的，而是靠状态流转推进的。

---

## 4.5 Verification Runtime：没有验证闭环，Agent 只是会行动的 Chatbot

这是 Agent Runtime 里最关键的一层。

Agent 不能只说“我完成了”。

它必须证明自己完成了。

在 Coding 场景里，验证可以包括：

```text
代码能否编译？
测试是否通过？
接口是否返回正确？
页面是否正常渲染？
SQL 是否执行成功？
文件是否真的生成？
性能是否退化？
安全边界是否被破坏？
```

没有验证，所谓完成只是模型的一种表达。

很多 AI Coding 工具的问题不在于不会生成，而在于生成之后缺少可靠验证。

更可靠的闭环应该是：

```text
Generate → Verify → Repair
```

而不是：

```text
Generate → Done
```

这也是代码领域特别适合 Agent 化的原因。

因为代码有相对明确的反馈机制。

能不能编译，测试能不能通过，接口返回是否符合预期，这些都可以被工具验证。

一旦有了验证，Agent 就可以根据结果继续修复。

这也是 Karpathy 在 Agentic Engineering 里强调质量、安全和可维护性的原因。Agentic Engineering 不是盲目接受生成结果，而是要写规格、监督计划、检查 diff、编写测试、创建评估循环、管理权限，并保持质量。

所以我认为：

> **Agentic Engineering 的核心不是生成，而是验证后的生成。**

更直接一点：

> **没有 Verifier 的 Agent，本质上只是一个会行动的 Chatbot。**

Verifier 才是 Agent 从“能做事”走向“可信任”的关键。

---

## 4.6 Governance Runtime：Agent 越能干，越需要边界

最后一层是 Governance Runtime，也就是权限、边界和治理。

很多人做 Agent 产品时，容易本能地追求更强能力：

```text
能不能自动改代码？
能不能自动跑命令？
能不能自动部署？
能不能自动处理线上问题？
```

但企业级 Agent 不能只看能力，还要看边界。

它需要明确：

```text
哪些操作可以自动执行？
哪些操作必须人工确认？
哪些文件不能修改？
哪些命令不能运行？
哪些数据不能进入上下文？
失败多少次必须停止？
如何记录审计日志？
如何回滚？
如何追责？
```

在个人项目里，Agent 出错可能只是生成一堆烂代码。

但在企业系统里，Agent 出错可能意味着：

```text
误删数据
泄露信息
错误发布
破坏兼容性
污染代码库
引入安全漏洞
```

所以，Agent 的落地不是追求无限自主，而是追求边界内自主。

这也是 Agent Runtime 最容易被忽略，却最重要的部分之一。

因为模型能力决定的是 Agent 的上限。

治理机制决定的是 Agent 能不能真正进入生产环境。

所以：

> **Agent 的上限由模型能力决定，但 Agent 的落地能力由治理机制决定。**

---

# 05 为什么单 Agent 很难成为最终答案？

理解了 Agent Runtime，再回头看单 Agent，就会更清楚它的问题。

很多 Agent 项目早期都会从单 Agent 开始。

一个 Agent 接收需求，然后自己规划、自己执行、自己调用工具、自己写代码、自己检查结果、自己总结记忆。

这个结构很简单：

```text
User
 ↓
Big Agent
 ↓
Tools
```

在 demo 阶段，它很有效。

短任务、简单任务、边界清晰的任务，一个强模型加几个工具，确实可以做出很惊艳的效果。

但复杂工程任务里，单 Agent 很容易遇到四类问题。

第一，上下文污染。

所有需求、历史对话、错误日志、工具返回、中间计划、失败尝试都堆在同一个上下文里。上下文越长，不一定越聪明，反而可能越混乱。

第二，角色冲突。

规划者、执行者、验证者本来应该有不同立场。

规划者要全局，执行者要落地，验证者要怀疑。

如果一个 Agent 自己规划、自己执行、自己验证、自己宣布完成，这在工程上并不可靠。

第三，长任务稳定性差。

短任务可以靠模型能力硬顶，但长任务需要状态、记忆、验证、恢复和边界。

没有这些机制，Agent 很容易目标漂移、重复尝试、忘记约束，或者在错误方向上越走越远。

第四，不可观测、不可调试。

当一个 Agent 什么都做时，一旦结果出错，你很难判断到底是哪里出了问题。

是需求理解错了？

是计划拆解错了？

是工具调用错了？

是代码生成错了？

是验证不充分？

还是记忆污染了？

所以，单 Agent 的问题不是不够智能，而是：

> **不可分解、不可观测、不可治理。**

这也是为什么复杂工程任务最终不会停留在“一个大 Agent + 一堆工具”的结构上，而会走向更清晰的 Runtime Engineering。

未来更可靠的 Agent 系统，很可能是多个角色、多个状态、多个验证器、多个工具边界共同组成的系统。

模型是核心，但不是全部。

真正重要的是模型周围的工程结构。

# 06 程序员的价值会迁移到哪里？

如果 AI 真的越来越会写代码，那程序员的价值会迁移到哪里？

我觉得至少会迁移到五件事上。

## 第一，定义问题

AI 可以帮你实现目标，但它不能替你判断目标是否值得做。

很多时候，真正难的不是写代码，而是判断：

```text
这个需求是不是伪需求？
这个问题是不是应该用技术解决？
这个功能是不是会增加系统复杂度？
这个产品方向是不是值得投入？
```

AI 很适合回答“怎么做”。

但“该不该做”，仍然需要人判断。

## 第二，设计系统

AI 可以生成模块，但系统边界、抽象层次、数据流、权限模型、长期演进路径，仍然需要人判断。

尤其是复杂系统里，真正影响长期质量的往往不是某一段代码，而是结构。

比如：

```text
模块怎么拆？
状态归谁管理？
数据一致性怎么保证？
接口如何兼容？
失败后怎么恢复？
权限边界怎么设计？
```

这些问题不能完全交给 AI 自由发挥。

AI 可以参与讨论，可以给方案，但最终判断要由人负责。

## 第三，组织上下文

未来会用 AI 的人很多。

但能把复杂背景、历史约束、项目事实、代码结构、用户偏好组织成高质量上下文的人，不会太多。

这会成为非常重要的能力。

同样一个任务，有人只会说：

```text
帮我优化一下这个接口。
```

另一个人会说：

```text
这个接口服务高频查询，当前瓶颈在分页和索引。
不要改返回结构，因为前端多个页面依赖它。
可以调整 SQL 和索引，但不能引入新缓存。
完成后需要跑这三个测试，并解释执行计划变化。
```

这两种输入，得到的结果一定不一样。

未来工程师不是不写代码了，而是越来越多地写上下文、约束和验收标准。

## 第四，建立验证

AI 生成结果之后，如何证明它是对的，会成为工程师非常重要的能力。

你要知道：

```text
应该写什么测试？
应该检查什么指标？
应该观察什么日志？
应该压测什么路径？
应该验证什么边界？
应该让哪些 Agent 互相攻击和检查？
```

AI Coding 的核心不是“让 AI 写完”，而是“让 AI 写完之后能被验证”。

这会让测试、评估、自动化验证、灰度、回滚这些能力变得更重要。

## 第五，设计 Runtime

最后，也是最关键的能力：设计 Runtime。

也就是让 Agent 在一个可控、可靠、可恢复的环境中完成任务。

这包括：

```text
Context 怎么组织
Tool 怎么封装
Memory 怎么沉淀
State 怎么流转
Verifier 怎么设计
权限怎么控制
失败怎么恢复
人什么时候介入
```

这就是未来程序员新的壁垒。

所以我认为：

> **代码会越来越便宜，但判断力会越来越贵。**
>
> **AI 可以替你写代码，但不能替你承担工程判断的责任。**

# 07 未来优秀工程师会变成 Agentic System Engineer

我不认为未来优秀工程师会简单变成 Prompt Engineer。

Prompt Engineer 这个词太窄了。

它让人误以为未来的工作只是写好提示词。

但真正有价值的能力，不只是写一句漂亮 Prompt，而是设计一个智能执行系统。

未来优秀工程师需要理解：

```text
Agent 如何拆分任务？
Context 如何组织？
Tool 如何封装？
Memory 如何沉淀？
State 如何流转？
Verifier 如何设计？
人应该在什么时候介入？
系统失败后如何恢复？
权限边界如何治理？
```

这已经不是单纯的 Prompt 问题，而是系统设计问题。

所以我更愿意把这类工程师称为：

> **Agentic System Engineer**

也就是能够设计、调度和治理 Agent 系统的人。

过去，一个优秀工程师的能力模型可能是：

```text
语言熟练度
框架经验
数据库能力
系统设计
性能优化
工程协作
```

这些能力仍然重要。

但在 Agent 时代，它们会叠加一组新能力：

```text
Prompt 设计能力
Context 组织能力
Agent 拆分能力
Tool 封装能力
Memory 设计能力
State Machine 设计能力
Verifier 设计能力
Human-in-the-loop 设计能力
Runtime Debug 能力
```

这不是让工程师变得更轻松，而是让工程师的工作重心上移。

你不再只是写代码，而是在设计一个能让 AI 安全、高效工作的系统。

所以：

> **会用 AI 写代码，只是入门；会设计 Agent Runtime，才是壁垒。**

这也是未来优秀工程师和普通 AI 使用者之间的分水岭。

普通使用者让 AI 生成结果。

优秀工程师设计系统，让 AI 持续、稳定、可验证地产生结果。

# 08 结尾：AI 不会立刻淘汰程序员，但会重新给程序员定价

回到最开始的问题：

AI 会写代码，程序员是不是危险了？

我的答案是：

危险，但不是所有程序员都危险。

真正危险的是那些只把自己定位成“代码实现者”的程序员。

因为实现会越来越便宜，代码会越来越容易生成，很多过去需要手写的工作都会被 AI 压缩。

但另一类程序员会变得更重要。

他们能定义问题，设计系统，判断取舍，建立验证，封装工具，设计 Agent Runtime，让 AI 在真实工程环境中稳定工作。

这才是 Karpathy 这次访谈真正给我的启发。

Vibe Coding 不是终点。

它只是让我们第一次直观感受到：自然语言可以驱动软件生成。

但接下来真正重要的，是 Agentic Engineering。

而 Agentic Engineering 真正的壁垒，不是模型多会写代码，而是 Runtime 能不能让 Agent 可控、可靠、可验证。

所以，AI 会写代码不可怕。

真正可怕的是：

> **当 AI 已经会写代码之后，你还只会写代码。**

未来的软件工程，不会简单变成“人不写代码，AI 写代码”。

更可能变成：

```text
人定义目标
人设计边界
人组织上下文
人封装工具
人建立验证
人治理 Agent
AI 执行大量具体实现
```

这意味着程序员不会消失。

但程序员会分化。

只会写代码的人，会越来越容易被工具压缩空间。

能设计系统、判断方向、构建 Agent Runtime 的人，会越来越重要。

因为在 AI Coding 时代，真正稀缺的不是代码本身，而是：

```text
判断力
系统设计能力
工程约束能力
验证能力
以及对复杂系统的掌控感
```

这也许才是 AI Coding 对程序员价值最大的重估。
